Harden auth: retry-on-401 with forced refresh, not just expiry prediction

While re-verifying the previous fix, found the expiry-check approach in
getAuthToken() isn't sufficient on its own. Live evidence: pulled the actual
token out of this browser's localStorage mid-session and it had 9 minutes
left on its own `exp` claim, but /api/screenings still rejected it outright
with 401 the moment it was used. Confirmed by hitting the API directly with
that exact token via the browser console -- not a caching or deploy-lag
artefact.

This means a token can go bad before its own claimed expiry (revoked
elsewhere, signing key rotated, anything server-side) -- something a purely
local "is expires_at close?" check can never predict, because it never asks
the server. The only reliable fix is to react to the server's actual answer.

Added authedFetch() to lib/getAuthToken.js: attaches a token via the existing
getAuthToken(), and if the response comes back 401, forces a real
supabase.auth.refreshSession() and retries exactly once before handing the
final response back to the caller.

Replaced every fetch+manual-Bearer-header call in the two files most
directly implicated in this conversation with authedFetch():
- components/EventSlideOut.js -- Coordinator load(), patchAction() (covers
  every Coordinator action: cancel, add walk-up booking, refunds, payment
  toggles/reminders, notes/description saves), and the resident-facing
  handleBook/handleModify/handleCancel/handleMarkSubmitted. Both now-unused
  local getToken() wrappers removed.
- app/(app)/screenings/page.js -- list load, slide-out refresh, add/edit
  screening form (the exact path that went silently empty).

Not yet extended to Social/Book Club/Admin's remaining call sites (still on
the plain getAuthToken() fix from the previous commit) -- flagging as a
known next step if the same symptom turns up there, rather than pushing a
larger unverified sweep in one go.

Verified: npm run build clean.
