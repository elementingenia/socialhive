Fix Coordinator View 'Unauthenticated' error on stale mobile sessions

Reported: EC/admin opening Coordinator View on the Jerry & Marge screening
saw "Coordinator view unavailable: Unauthenticated" on an iPhone. Could not
reproduce with a fresh Chrome login as Iain -- Coordinator View, attendee
list, and the Cancel Booking dialog (verified separately, see prior portal
fix) all loaded correctly. That rules out a broken backend or a regression
in the portal fix, and points at something specific to a long-lived mobile
session instead.

Root cause found in code: EventSlideOut.js's two getToken() helpers (Coordinator
panel + booking actions) call supabase.auth.getSession() and hand the result
straight to the API with no expiry check. supabase-js's background
auto-refresh timer only runs while the page's JS is alive -- mobile Safari
freezes JS timers on a backgrounded/locked tab, so a session left open for a
while (very plausible for an EC checking back on an event later) can sit with
a dead access token. getSession() just returns that stale token synchronously,
the API correctly rejects it as Unauthenticated, and the UI had no recovery
beyond a manual Retry link.

Fix: new lib/getAuthToken.js checks the session's expires_at and forces
supabase.auth.refreshSession() first if it's expired or expiring within 60s,
before returning a token. Both getToken() functions in EventSlideOut.js now
delegate to it. Same stale-token pattern exists in ResidentEditPanel.js's
exported getToken() (used by Admin/Contacts/Documents) and
social/events/page.js -- not touched in this pass since neither was reported
as broken, flagging as a known follow-up rather than widening scope on
unverified code paths.

Verified: npm run build clean. This is the same class of bug as the Cancel
Booking fix -- a real device/session-state condition I can't force from a
fresh desktop Chrome login, so please confirm on your phone next time
Coordinator View has sat open/backgrounded for a while.
