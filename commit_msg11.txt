Social reconciliation: add refunds tracking, staleness flag, drop "Reconciled" wording

Iain's Bastille Day critique (2026-07-14), three concrete asks -- all three built:

1. Refunds Due / Refunds Issued for Social. Confirmed via code: Movies and
   Book Club's Coordinator panel (components/EventSlideOut.js) already had
   this via the set_refund action, Social never got it -- zero refund
   tracking anywhere on this hub. A cancelled-but-paid booking just
   vanished; paymentSummary() explicitly skipped refunded rows and never
   saw refund-*pending* rows at all (those were never passed to it).
   lib/payments.js's paymentSummary() now takes a third argument
   (refundPendingBookings) and returns refundsDueTotal/refundsDueCount,
   rendered as its own line in the summary. Added the same Refunds Due
   (amber, "Mark Refunded") / Refunds Issued (grey, "Unmark") sections
   Movies/Book Club have, reusing the exact same set_refund action --
   no backend changes needed, it already existed and already worked, Social
   just never called it. Live-fire verified end-to-end against production
   (disposable event + testbot): cancelled a paid booking -> correctly
   landed in Refunds Due -> marked refunded -> correctly moved to Refunds
   Issued. Cleaned up afterward.

2. Staleness flag. New lib/payments.js reconciliationIsStale(event,
   allBookings) compares payments_reconciled_at against the latest booking
   activity and flags it inline on the same line as the review stamp
   ("... -- new activity since, worth another look") instead of a separate
   banner, keeping the vertical-space-conscious layout intact. Verified the
   full summary math (expected/collected/outstanding/refunds due) against
   Bastille Day's actual live numbers reproduced in a Node test:
   $350/$210/$140/$70 -- matches exactly, plus correctly surfaces the $70
   refund that was previously invisible.

   Known limitation, disclosed not hidden: today this only catches new
   bookings added after the reconciliation stamp, via booked_at -- it can't
   yet catch a plain cancellation-after-reconciliation (no timestamp exists
   for that). supabase/migrations/040_bookings_updated_at.sql adds the
   column needed to close that gap fully, but the code deliberately does
   NOT select or write it yet -- confirmed live that column doesn't exist
   in production, and referencing it in a SELECT would 400 the entire
   Social events query for everyone. This migration needs to be run
   manually in the Supabase SQL editor; once it is, a small follow-up
   (adding updated_at to the bookings select + a few UPDATE calls) turns on
   full staleness detection with no further design change needed.

3. Renamed "Reconciled" -> "Last reviewed" on the summary line. Close Out
   was always meant to be re-runnable (migration 037's own comment says so
   explicitly), never a lock -- "Reconciled" implied a finality the feature
   never had, which is exactly what made a stale stamp next to live-updated
   numbers read as contradictory instead of just "due for another check."

Verified: npm run build clean. Payment math cross-checked against real
production numbers via a standalone Node script (not committed).
